## 🩺 최초 진단 (강점 / 약점)
*2025년 첫 진단 기준*

| 영역 | 상태 | 비고 |
|---|---|---|
| 자료구조 / Big-O | 🟡 직관은 있으나 용어·표기 없음 | 해시맵 검색이 빠른 이유는 알지만 O(n)/O(1)로 설명 못 함 |
| REST / HTTP 원리 | 🔴 표면만 | url vs body는 알지만 멱등성·안전성 모름 |
| DB 설계 | 🟢 탄탄 | 1:N 관계, FK 정확히 이해 |
| 프론트 상태 관리 | 🟢 감각 좋음 | 로딩/성공/실패 인지, empty/refetch만 보강 |
| LLM / RAG | 🟡 직관 있음, 경험 없음 | "문서를 기억" 직관 → 임베딩/벡터DB 개념으로 채울 것 |
| Git 워크플로우 | 🔴 기본만 | add/commit/push만. branch·PR·CI 모름 |

**한 줄 진단:** 만들 줄은 안다. 구멍은 "원리를 설명하는 언어"와 "협업 흐름"에 있다.

## ❌ 오답노트 (Error Log) — 에이전트가 간격 반복으로 되묻는 항목
> 형식: 주제 | 무엇을 틀렸나 | 정답 한 줄 | 복습 간격 | 다음 복습일

1. **Big-O 표기** | 배열 검색이 느린 이유를 "주소 찾기"로 설명함 | 값 검색 시 배열은 O(n), 해시맵은 O(1) | 1일 | (첫 세션 다음날)
2. **HTTP 멱등성/안전성** | GET/POST를 url vs body로만 설명 | GET=safe(상태 불변)·여러 번 호출해도 동일, POST=호출마다 새 리소스 생성 | 1일 | (첫 세션 다음날)
3. **Git 협업 흐름** | main 직접 push가 왜 위험한지 "다같이 써서"까지만 | 검증 안 된 코드가 main에 들어가면 전원 작업·배포가 깨짐 → branch→PR→리뷰/CI→머지 | 1일 | (첫 세션 다음날)
4. **RAG 구조** | "문서를 기억" 수준의 직관만 | 문서를 청크로 쪼개 임베딩→벡터DB 저장, 질문 시 관련 청크만 검색해 LLM에 함께 전달 | 3일 | (첫 세션 +3일) — 프로젝트로 자연 학습 예정
